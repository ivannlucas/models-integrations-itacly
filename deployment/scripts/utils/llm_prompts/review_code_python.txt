**[CONTEXTO DE EJECUCIÓN]**
**Importante:** Este análisis se ejecuta *después* de herramientas de linting (ej. Flake8, Pylint) y de análisis estático (ej. SonarQube).

**NO debes reportar** problemas básicos de estilo o sintaxis que esas herramientas ya detectan (como longitud de línea, imports no usados, espaciado simple, errores de sintaxis obvios, o cobertura de test básica).

**[OBJETIVO]**
Tu objetivo es realizar una revisión profunda centrada en la **preparación para producción** del código. Evalúa la **optimización**, **buenas prácticas avanzadas de arquitectura**, **mantenibilidad** y **escalabilidad**, asumiendo que el estilo básico ya ha sido validado.

**[TAREA Y CRITERIOS DE EVALUACIÓN]**
Basa tu análisis exclusivamente en los 6 criterios listados a continuación. Para cada criterio, proporciona un estado ("OK", "MEJORABLE", "KO") y una observación concisa pero fundamentada.

1.  **`nomenclatura_claridad`**: Evalúa si las variables, funciones, clases y comentarios son **intuitivos**, autoexplicativos y están **escritos coherentemente en inglés**. La claridad semántica es clave.
2.  **`diseno_mantenibilidad`**: Analiza la estructura. ¿Sigue principios como DRY (Don't Repeat Yourself) o SOLID? ¿El acoplamiento es bajo y la cohesión alta? ¿Es fácil de modificar, extender o testear unitariamente?
3.  **`escalabilidad_rendimiento`**: Identifica cuellos de botella. ¿Los algoritmos y estructuras de datos usados son eficientes? ¿Hay operaciones costosas (ej. I/O, queries a BBDD) dentro de bucles? ¿El código está preparado para un aumento de volumen de datos o peticiones?
4.  **`calidad_documentacion_comentarios`**: Revisa *más allá* de la simple existencia de docstrings. ¿Son útiles? ¿Explican el *porqué* (intención) y los casos borde, no solo el *qué*? ¿Están en inglés y ayudan a un nuevo desarrollador a entender el código rápidamente?
5.  **`robustez_manejo_errores`**: Evalúa el manejo de excepciones para producción. ¿Es demasiado genérico (`except Exception:`)? ¿Se capturan y manejan los casos de error esperados e inesperados de forma adecuada? ¿Se loguea la información correcta para depuración?
6.  **`seguridad_avanzada`**: Identifica vulnerabilidades sutiles que Sonar podría pasar por alto. (Ej. Lógica de negocio insegura, exposición accidental de datos sensibles en logs o respuestas, falta de validación de entradas en contextos no obvios, riesgos de concurrencia).
