**[OBJETIVO]**
Analizar el script proporcionado y devolver un objeto JSON con tres campos:
1.  `classification`: La categoría más específica del script según las reglas.
2.  `key_dependencies`: Un array de strings con las bibliotecas *clave* detectadas (ej. `["pandas", "fastapi", "sklearn"]`).

**[REGLAS DE CLASIFICACIÓN (Datos, IA y Analítica)]**
Debes usar la categoría *más específica* posible:

* **`ModelInference_API`**: Si importa `fastapi`, `flask`, `starlette` Y TAMBIÉN bibliotecas de ML como `joblib`, `onnx`, `tensorflow`, `torch` o `sklearn` (para cargar modelos y servir predicciones).
* **`WebService_General`**: Si importa `fastapi`, `flask`, `starlette`, `django` y define rutas, pero *no* parece ser para inferencia de modelos.
* **`ModelTraining_Script`**: Si importa `sklearn`, `tensorflow`, `pytorch`, `xgboost`, etc., y contiene funciones como `.fit()`, `.train()`, o guarda artefactos (ej. `joblib.dump`).
* **`DataPipeline_ETL`**: Si realiza operaciones claras de E-T-L, importando `pandas`, `pyspark`, `sqlalchemy`, `polars`, `dask` y se enfoca en la lectura, transformación y carga de datos.
* **`Dashboard_App`**: Si importa `streamlit` o `dash` y define un layout de UI.
* **`DataAnalysis_Report`**: Si usa `pandas`, `matplotlib`, `seaborn`, `plotly` principalmente para análisis y visualización, pero no es un Pipeline ni un Dashboard.
* **`Utility_Library`**: Si define principalmente funciones de ayuda, clases de configuración, conectores, y no encaja en las anteriores.

**[REGLAS DE GENERACIÓN DE MARKDOWN]**
La documentación generada (`documentation`) DEBE seguir esta estructura. NO inventes información; usa marcadores como `[DESCRIPCIÓN PENDIENTE]` o `[TBD]` si el script no proporciona la información.

1.  **Título:** El nombre del fichero (ej. `Script: mi_script.py`).
2.  **Resumen y Objetivo:** Una descripción de alto nivel de *qué* hace el script y *por qué* existe (el problema de negocio que resuelve).
3.  **Arquitectura y Lógica Principal:** Una explicación de *cómo* funciona, idealmente en pasos o bullet points (ej. "1. Conecta a S3 -> 2. Aplica transformaciones X -> 3. Carga en Redshift").
4.  **Configuración Requerida:** Lista de las variables de entorno (`.env`) o argumentos de `argparse` que el script necesita para funcionar.
5.  **Entradas y Salidas:** Descripción de lo que consume (ej. "Tabla `users` de la BBDD X", "Payload JSON de la API") y lo que produce (ej. "Fichero Parquet en S3", "Modelo `model.pkl`").
6.  **Ejemplos de Uso:**
    * Si es un script (`ModelTraining_Script`, `DataPipeline_ETL`), infiere un comando de terminal (ej. `python train_model.py --data_path ./data/raw`).
    * Si es una API (`ModelInference_API`, `WebService_General`), genera un ejemplo de `curl` o un snippet de `requests` de Python con un payload de ejemplo.
    * Si es una librería (`Utility_Library`), muestra cómo importar y usar una función clave.
7.  **Mantenimiento y Puntos Clave:** Menciona cualquier "punto de falla" obvio y aspectos a tener el cuenta al utilizar o evolucionar el código (ej. "Depende de una API externa que falla a menudo", "El rendimiento puede degradarse si la tabla X crece").
