**[OBJETIVO]**
Tu objetivo es realizar una revisión profunda centrada en la **preparación para producción** del script SQL. Evalúa la **optimización**, **robustez transaccional**, **mantenibilidad** y **seguridad**, asumiendo que el formato básico ya ha sido validado.

**[TAREA Y CRITERIOS DE EVALUACIÓN]**
Basa tu análisis exclusivamente en los 6 criterios listados a continuación. Para cada criterio, proporciona un estado ("OK", "MEJORABLE", "KO") y una observación concisa pero fundamentada.

1.  **`nomenclatura_claridad_idioma`**: Evalúa si los alias, variables y comentarios son **intuitivos**, autoexplicativos y están **escritos coherentemente en inglés**.
2.  **`optimizacion_plan_ejecucion`**: Analiza el rendimiento. ¿La query es **SARGable** (apta para usar índices)? ¿Evita `SELECT *`? ¿El uso de `JOINs`, `WHERE` y `GROUP BY` es eficiente? ¿Existen funciones en el `WHERE` que impidan usar índices?
3.  **`mantenibilidad_legibilidad_sql`**: Evalúa la estructura. ¿Usa CTEs (Common Table Expressions) para clarificar sub-consultas complejas? ¿Evita anti-patrones (ej. joins cartesianos implícitos)? ¿La lógica es fácil de seguir para otro desarrollador?
4.  **`calidad_documentacion_comentarios`**: Revisa *más allá* de la simple existencia de comentarios. ¿Son útiles? ¿Explican la *intención* de negocio de un filtro complejo, no solo lo que hace la sintaxis? ¿Están en inglés?
5.  **`robustez_transaccional`**: (Aplicable a DML o scripts) ¿El script maneja las transacciones adecuadamente (`BEGIN TRAN`, `COMMIT`, `ROLLBACK`)? ¿Es atómico? ¿Contempla el manejo de errores (ej. `TRY...CATCH` en T-SQL)?
6.  **`seguridad_vulnerabilidades`**: Identifica riesgos. ¿Es vulnerable a Inyección SQL (especialmente en SQL dinámico)? ¿Expone datos sensibles? ¿Revela información de estructura en los errores?
